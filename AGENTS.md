# get-work — agent guide

Outreach toolkit for prospecting holiday-lighting software work. CLI: `./getwork` (see `./getwork --help`).

## Market research (pick where to prospect)

- `data/markets.json` is the target-market list (tiers, notes). Tier 1 = work first; tier 0 = Kansas City = NEVER prospect (Stetson's home market).
- `./getwork market-scan "Tulsa, OK"` searches DuckDuckGo, filters directories/lead-gen junk, verifies each domain's homepage mentions holiday lighting, saves `data/scans/<slug>.json`, and updates the market's installer count.
- `./getwork markets` regenerates `outreach/markets.html` (linked from the preview index). Never edit markets.html by hand.
- Primary search engine is the **Serper.dev API** (`SERPER_API_KEY` in `.env`, never committed) — fast, not IP-throttled. Fallbacks: Brave HTML (~1 query/min from this datacenter IP), then DDG. Last-resort fallback: **local-search handoff** — `data/inbox/<metro-slug>.txt` files contain the queries per metro; Stetson runs them from his local machine, pastes result URLs under each `q:` line, pushes, then we run `./getwork market-scan "Phoenix, AZ" --from-urls data/inbox/phoenix-az.txt`. Filtering/verification/saving all happen here as usual. `--print-queries` prints the queries for any metro.
- A scan is validation only — it builds no folders. Review the candidate list with Stetson before building prospects. Lawn-care companies that also hang lights are legitimate prospects (classic industry pattern).

## Prospecting workflow (the big one)

When asked something like *"find holiday lighting companies in Tulsa, OK, build their prospect folders, and update the summary"*, follow `docs/agent-prospecting.md`. Short version:

1. **Search the web** for holiday/Christmas light installation companies in the target area. Skip national franchises' corporate sites; prefer local operators with their own websites. Skip companies already in `outreach/prospects/`.
2. **Scrape** each candidate: `./getwork scrape <url>` (add `--json` for machine-readable). Verify it's a real local company.
3. **Build the folder**: `./getwork build-prospect <slug> "Display Name" <url> --location "City, ST"`. This scrapes, scaffolds `outreach/prospects/<slug>/` (portfolio.html, email.md, notes.md, scrape.json), fills notes.md with contacts, and regenerates summary + index.
4. **Enrich notes.md** by hand: residential/commercial focus, personalization lines, anything the scraper missed (check `scrape.json` and the live site).
5. **Draft the email**: write `outreach/prospects/<slug>/email-draft.md` (follow existing prospects like `holiglows/email-draft.md` for voice/format; positioning lives in `docs/holiday-lighting-handoff.md`). Set `Email drafted: YYYY-MM-DD` in the notes.md Send log.
6. **Update summary**: `./getwork summary` (also run automatically by build-prospect/index).

## Summary table

`outreach/summary.html` is generated by `scripts/generate-summary.py` from each prospect's `notes.md`. Never edit summary.html by hand. Fields read: H1 (name), `**Status:**`, `**Website:**`, `Contact person/email`, `Location / service area`, `Email drafted:`, `Email sent:`. Stetson fills in `Email sent:` manually after sending, then reruns `getwork summary`.

Statuses: drafting → ready → sent → replied (or dead).

## Conventions

- Slugs: lowercase, no spaces, usually the domain name without TLD (e.g. `holiglows`).
- Don't overwrite an existing prospect folder without `--force` and a good reason.
- Preview: `make dev` (BrowserSync, port 8765). Directory page links to the summary table.
- Commit after building prospects.
